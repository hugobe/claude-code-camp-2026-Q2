# Explore Agent Architectures

Choosing the right agent solution is often a major source of confusion for tech professionals, as the responsibilities of various options frequently seem to overlap.

To identify the most suitable solution for our workload, we will evaluate several agent architectures.

## 1. An agent file with referenced files eg. AGENT.md, @~/docs/*.md

The simplest agent is creating an "agent file" and possibly importing other files that are read conditionally when needed. 

We should attempt to create an agent file and see if it can connect to the mud and complete a simple goal. Example:  "Find the bakery and list the menu" or "Gather 100 coins and buy a sword".

We want to use the smallest and least intelligent model and scale up. 

###  Technical Observations:
 
We created a CLAUDE.md/AGENT.md with a simple prompt and told it will need to manage it its own local memory via simple markdown files. We provided it with the location of the MUD and the player credentials. 

#### Claude Haiku Medium
- It is very inefficient because it connects, runs the bash script with some commands, disconnects in a loop. It randomly tries different paths to get to the bakery.
- It repeatedly executed similar commands so that the player moved in a circle.
- Didn't update the player.md and world.md after executing scripts, I had to cancel the execution because it made no progress, therefore it also wasn't able to update afterwards.

#### Claude Haiku xHigh
- Only searched for files containing "world" or "bakery".
- Got stuck in the same cycle as Claude Haiku Medium, trying new ways but not reaching the goal.

#### Gemini 3.5 Flash (LOW)
- The agent generated Implementation Plans which always needed an approval to start.
- Easier goals were accomplished fast, but it struggled with more complex goals. 
- Recorded information in the player.md and world.md but they are quite cluttered and complex to understand.
- Instead of real playing to reach a goal like gather 100 coins it creates bots which run till the goal is accomplished. Unfortunately such bot-scripts can't react to new situations such as the player being killed by another mob.

#### General Observations
- None was able to manage the socket connection and handle the core connection issue.
- Each one created bash or python scripts to interact with the MUD
- MUDs need agents that can react directly to new events, a fixed script can't provide that.
- Each execution needs to be checked and approved manually.

### Technical Conclusion
A coding harness alone is not the best fit for this architecture, it needs for every action a script, which forces the model to regenerate a the same login and connection function on every attempt instead of reusing it.

The login and connection handling should be solved in reusable programm, a dedicated MUD SDK or MUD Manager, rather than being regenerated by the model each time. This would remove the repeated connect, parse and disconnect cycle and give a responsive interface to build on.

An MCP server could make that SDK's functions available to the agent as tools, so the he can interact with the session directly instead of spawning scripts and enabling a specialized agent loop with better reliability, visibility, and token efficiency.

Markdown files may also not be sufficient once player and world state grows plain text doesn't hold structured and gets cluttered fast, as already seen with Gemini's output.

## 2. Agent Skills driven by main agent eg. ~/.skills

A very common way to drive specific functionality is via Agent Skills, an open format for agents adopted by many coding harnesses and agent SDKs.

### Technical Observation

Creating an skill with the official claude creator skill was successful. It was able to create a reliable connection to the mud where our agent was able to play the game responsively.

While the agent was able to complete both simple and more complex goals, it required a huge amount of calls. It tended to execute tasks immediately, attempting advanced objectives that the character was unable to handle without first breaking them down into smaller, manageable sub-tasks. 

When given the broader task of defeating the massive Minotaur in the newbie zone, the agent eventually located the zone after a long search and successfully figured out how to open locked doors. However, upon finding the Minotaur, it attacked immediately without considering whether the character was strong enough, which led to the character's death.

Compared to a real player, the agent only tries to achieve the current goal, it doesn't consider killing other mobs or tasks on its way to level up.

It updated the data/world.md and data/player.md only after it completed a goal and not while playing to keep track of the world or player stats.

Claude' Code's agentic loop is a good driver but if they were to update Claude Code we would have bo idea how the agentic loop would be affected.

I could see it having a hard time managing the state of just markdown files for memory. Also inside the World.md it creates different headlines for different topics e.g. World, Shops, Guilds. It would be more efficient to create different files for each topic so that it doesn't have to read whole file to find informations. If they grew too large. 

I think we need dynamic adaptive task management:

e.g. defeat the massive minotaur in the newbie zone north of town

Before I find the Newbie Zone and leave the town, do I need to prepare?
- Collect information from NPCs for my goal.
- Can I obtain any resources? 
- Any training I need to do?

I should find the Newbie Zone.
- While on path, was there anything of interest that should warrant a detour? Would this spawn a side quest? 
- Explorer mode: 
    - Focused: Stay on my main quest
    - Curious: Consider side quest right on main quest especially if could save backtracking or provide an advantage or resources 
    - Aloof: Do all sidequests and not worry too quickly about main quest progression. 

I have found the Newbie Zone.
- Risk Mode: 
    - Bold: Prioritize aggressive exploration to locate the primary objective quickly. Attempt to run past high-level threats or flee when necessary, and actively engage tougher enemies for faster leveling, accepting higher risk.
    - Balanced: Balance safety and speed, taking calculated risks when the payoff is high but prioritizing survival.
    - Cautious / Risk-Averse: Avoid advancing into areas with higher-level monsters where death is a real threat. Prioritize staying in safe zones to recover health. Frequently return to town to restock resources, manage hunger/thirst, and secure gold. (Note: Threat assessment must be contextual—harmless high-level NPCs like town guards should be distinguished from aggressive monsters in the wild).

To build a more capable and human-like player agent, we should define a Player Persona and implement a structured Goal Decomposition process:

- Guide the agent's behavior style using a mix of operational modes (e.g., combining the Explorer Mode and Risk Mode defined above). This ensures the agent acts consistently—whether playing conservatively to accumulate resources or playing boldly to speedrun goals.
- Before executing any complex objective (e.g., *"defeat the massive minotaur"*), the agent should automatically decompose it into a sequence of sub-goals to accomplish the main goal. This allows developers to audit and preview the agent's planned reasoning before it executes.

### Technical Conclusions

Utilizing Agent Skills is a reasonable approach, managing a full player journey requires more sophisticated state, world, and player representation. To scale this effectively, we need:
- Clear tracking of token consumption, cost reporting, and the ability to review the complete step-by-step player path.
- A dedicated loop for active execution, minimizing passive wait times where the agent is stuck prompting the user or asking for instructions on what to do next.
